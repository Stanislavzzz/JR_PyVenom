# Git и GitHub: базовые команды

---

## 1. Установка Git

Официальная страница установки Git:

https://git-scm.com/install/

### macOS

Установка Git через Homebrew:

```bash
brew install git
````

### Linux

Установка Git через пакетный менеджер `apt`:

```bash
sudo apt install git
```

---

## 2. Проверка установки Git

После установки проверяем, что Git доступен в терминале:

```bash
git --version
```

Команда покажет установленную версию Git.

---

## 3. Первичная настройка Git

Перед началом работы нужно указать имя и email автора коммитов.

```bash
git config --global user.name "Stanislav"
```

```bash
git config --global user.email "stanislav@gmail.com"
```

Проверить текущие глобальные настройки:

```bash
git config --global --list
```

---

## 4. Создание локального Git-репозитория

Переходим в папку проекта и инициализируем Git:

```bash
git init
```

После этого в проекте появится скрытая папка `.git`, где Git будет хранить историю изменений.

---

## 5. Файл `.gitignore`

Файл `.gitignore` нужен, чтобы указать Git, какие файлы и папки не нужно отслеживать.

Например:

```text
.venv/
__pycache__/
.env
.idea/
```

Готовые шаблоны `.gitignore` можно создать здесь:

[https://www.toptal.com/developers/gitignore](https://www.toptal.com/developers/gitignore)

---

## 6. Проверка состояния репозитория

Команда показывает, какие файлы изменены, добавлены или ещё не отслеживаются Git:

```bash
git status
```

---

## 7. Добавление файлов в индекс

Добавляем конкретный файл `.gitignore`:

```bash
git add ./.gitignore
```

Добавляем конкретный файл `README.md`:

```bash
git add ./README.md
```

---

## 8. Первый коммит проекта

Фиксируем добавленные файлы в истории Git:

```bash
git commit -m "Add project"
```

Коммит - это сохранённая точка состояния проекта.

---

## 9. Добавление всех изменений

Добавляем все изменённые и новые файлы:

```bash
git add .
```

Создаём новый коммит:

```bash
git commit -m "Add main.py"
```

---

## 10. Просмотр истории коммитов

Полная история коммитов:

```bash
git log
```

Краткая история в одну строку:

```bash
git log --oneline
```

История в виде графа:

```bash
git log --graph
```

---

## 11. Переход к старому коммиту

Можно временно перейти к конкретному коммиту по его хэшу:

```bash
git checkout 0307cf
```

Проверить текущие ветки:

```bash
git branch
```

Вернуться обратно в основную ветку:

```bash
git checkout main
```

---

## 12. Быстрый коммит изменённых файлов

Команда сразу добавляет уже отслеживаемые изменённые файлы и создаёт коммит:

```bash
git commit -am "edit readme"
```

Важно: эта команда не добавляет новые файлы, которые Git ещё не отслеживает.

---

## 13. Создание репозитория на GitHub

Переходим на сайт GitHub:

[https://github.com/](https://github.com/)

Создаём новый репозиторий, например:

```text
test_jr
```

---

## 14. Подключение локального проекта к GitHub

Добавляем удалённый репозиторий:

```bash
git remote add origin https://github.com/Stanislav/test_jr.git
```

Переименовываем текущую ветку в `main`:

```bash
git branch -M main
```

Отправляем проект на GitHub:

```bash
git push -u origin main
```

После этого локальный проект будет загружен в удалённый репозиторий GitHub.

---

## 15. Общая последовательность работы

Обычно работа с Git выглядит так:

```bash
git status
git add .
git commit -m "Описание изменений"
git push
```

То есть:

1. Проверяем состояние проекта.
2. Добавляем изменения.
3. Создаём коммит.
4. Отправляем изменения на GitHub.




## 16. Получение последних изменений с GitHub

Команда забирает свежие изменения из удалённого репозитория и сразу применяет их в текущей ветке:

```bash
git pull
````

Обычно используется перед началом работы, чтобы локальный проект был актуальным.

---

## 17. Клонирование готового репозитория

Команда скачивает проект с GitHub на компьютер:

```bash
git clone https://github.com/Stanislavzzz/MT5bot.git
```

После выполнения появится папка проекта `MT5bot`.

---

## 18. Создание новой ветки

Создаём новую ветку `develop`:

```bash
git branch develop
```

Важно: эта команда только создаёт ветку, но не переключает на неё.

---

## 19. Переключение на другую ветку

Переключаемся на ветку `new1`:

```bash
git checkout new1
```

Ветка `new1` уже должна существовать.

---

## 20. Создание новой ветки и переход в неё

Современный вариант команды:

```bash
git switch -c new3
```

Эта команда создаёт ветку `new3` и сразу переключает нас на неё.

---

## 21. Создание новой ветки и переход в неё через checkout

Классический вариант команды:

```bash
git checkout -b new4
```

Команда делает то же самое: создаёт ветку `new4` и сразу переключает нас на неё.

---

## 22. Переключение на ветку develop

Современный вариант переключения между ветками:

```bash
git switch develop
```

Команда переключает проект на ветку `develop`.

---

## 23. Слияние ветки с текущей веткой

Находясь в ветке `develop`, можно влить в неё изменения из ветки `new1`:

```bash
git merge new1
```

То есть изменения из `new1` попадут в текущую ветку.

---

## 24. Удаление ветки безопасным способом

Удаляем ветку `new1`:

```bash
git branch -d new1
```

Git удалит ветку только если её изменения уже были слиты.

---

## 25. Принудительное удаление ветки

Удаляем ветку `new3` принудительно:

```bash
git branch -D new3
```

Эта команда удаляет ветку даже если её изменения не были слиты.

---

## 26. Просмотр истории всех веток

Команда показывает историю коммитов в виде графа по всем веткам:

```bash
git log --graph --oneline --all
```

Удобно использовать, чтобы увидеть, как ветки расходились и сливались.

---

## 27. Отправка ветки develop на GitHub

Команда отправляет локальную ветку `develop` в удалённый репозиторий:

```bash
git push -u origin develop
```

Флаг `-u` связывает локальную ветку `develop` с удалённой веткой `origin/develop`.

После этого для этой ветки можно будет использовать просто:

```bash
git push
```

или:

```bash
git pull
```

---

## 28. Типовой сценарий работы с ветками

Пример обычной последовательности:

```bash
git pull
git switch -c new-feature
# вносим изменения в код
git add .
git commit -m "Add new feature"
git switch develop
git merge new-feature
git branch -d new-feature
git push
```

Смысл:

1. Обновили проект.
2. Создали отдельную ветку для задачи.
3. Внесли изменения.
4. Сделали коммит.
5. Вернулись в основную рабочую ветку.
6. Влили изменения.
7. Удалили временную ветку.
8. Отправили результат на GitHub.

## 30. Просмотр истории всех веток

Команда показывает историю коммитов в коротком виде, с графом и по всем веткам:

```bash
git log --oneline --graph --all
````

Что означает:

```text
--oneline   показать каждый коммит в одну строку
--graph     показать связи между ветками в виде графа
--all       показать историю всех веток
```

Удобно использовать, когда нужно увидеть, где ветки расходились и где были слияния.

---

## 31. Поиск коммитов по автору

Команда показывает только те коммиты, которые сделал конкретный автор:

```bash
git log --author="Stanislav"
```

Например, так можно посмотреть все коммиты пользователя `Stanislav`.

Важно: Git ищет автора по имени или email, которые были указаны в настройках:

```bash
git config --global user.name
git config --global user.email
```

---

## 32. Поиск коммитов за период времени

Команда показывает коммиты, сделанные в указанном промежутке времени:

```bash
git log --since="2026-05-01 19:00:00" --until="2026-05-01 19:20:00"
```

Что означает:

```text
--since   начиная с какого времени искать
--until   до какого времени искать
```

Так удобно искать изменения, которые были сделаны во время занятия или в конкретный рабочий промежуток.

---

## 33. Поиск коммитов по сообщению

Команда ищет коммиты, в сообщении которых есть указанное слово:

```bash
git log --grep="readme"
```

Например, команда найдёт коммиты с сообщениями вроде:

```text
Add readme
Update README.md
edit readme
```

Важно: поиск идёт по тексту сообщения коммита, а не по содержимому файлов.

---

## 34. История изменений конкретного файла

Команда показывает только те коммиты, в которых изменялся файл `README.md`:

```bash
git log -- README.md
```

Это удобно, когда нужно понять, кто и когда менял конкретный файл.

## 35. Что такое fork

`Fork` — это копия чужого репозитория в вашем GitHub-аккаунте.

Простыми словами:

```text
Чужой репозиторий → Fork → Ваша копия репозитория
```

Fork нужен, когда у вас нет прав напрямую пушить в чужой проект, но вы хотите предложить свои изменения.

---

## 36. Как делаем fork: схема

Общая схема работы через fork:

```text
1. Находим чужой репозиторий на GitHub

2. Нажимаем кнопку Fork

3. GitHub создаёт копию репозитория в вашем аккаунте

4. Клонируем свой fork на компьютер

5. Создаём отдельную ветку для задачи

6. Вносим изменения в код

7. Делаем commit

8. Делаем push в свой fork

9. Создаём Pull Request в исходный репозиторий
```

---

## 37. Пример работы с fork через команды

Клонируем свой fork:

```bash
git clone https://github.com/Stanislav/project.git
```

Переходим в папку проекта:

```bash
cd project
```

Создаём новую ветку для задачи:

```bash
git switch -c fix-readme
```

Вносим изменения в файлы.

Проверяем состояние:

```bash
git status
```

Добавляем изменения:

```bash
git add .
```

Создаём коммит:

```bash
git commit -m "Fix README"
```

Отправляем ветку в свой fork:

```bash
git push -u origin fix-readme
```

После этого на GitHub создаём Pull Request.

---

## 38. Что такое Pull Request

`Pull Request`, или `PR`, — это запрос на добавление ваших изменений в другой репозиторий или в другую ветку.

Простыми словами:

```text
Я сделал изменения в своей ветке.
Пожалуйста, проверьте их и добавьте в основную ветку проекта.
```

Чаще всего PR используют на GitHub.

---

## 39. Что такое Merge Request

`Merge Request`, или `MR`, — это почти то же самое, что Pull Request.

Разница в названии:

```text
GitHub    → Pull Request
GitLab    → Merge Request
```

Смысл один и тот же: предложить изменения и попросить влить их в другую ветку.

---

## 40. Как сделать PR из одной ветки в другую в своём репозитории

Допустим, есть две ветки:

```text
main
develop
```

И мы хотим влить изменения из `develop` в `main`.

Схема:

```text
develop → Pull Request → main
```

На GitHub:

```text
1. Открываем свой репозиторий

2. Переходим во вкладку Pull requests

3. Нажимаем New pull request

4. Выбираем base-ветку: main

5. Выбираем compare-ветку: develop

6. Проверяем изменения

7. Нажимаем Create pull request

8. После проверки нажимаем Merge pull request
```

---

## 41. Как сделать PR из fork-проекта

Допустим, есть исходный проект:

```text
Original repository
```

И есть ваш fork:

```text
Your fork
```

Схема:

```text
ваш fork / fix-readme → Pull Request → original repository / main
```

На GitHub:

```text
1. Открываем свой fork

2. Переходим во вкладку Pull requests

3. Нажимаем New pull request

4. Указываем, куда отправляем изменения:
   base repository: исходный проект
   base branch: main

5. Указываем, откуда берём изменения:
   head repository: ваш fork
   compare branch: ваша ветка

6. Проверяем список изменений

7. Нажимаем Create pull request
```

---

## 42. Важная разница между merge и PR

`git merge` — это локальное слияние веток через терминал:

```bash
git switch main
git merge develop
```

`Pull Request` — это слияние через GitHub с возможностью проверки кода:

```text
создали ветку → сделали push → открыли PR → проверили → влили изменения
```

В командной работе чаще используют PR/MR, потому что так удобнее проверять код перед слиянием.
